# AI 활용 이력

이 문서는 `member-dragon/` 작업에서 실제로 어떤 프롬프트를 주고, AI 응답 중 무엇을 채택했는지 흐름대로 정리한 기록이다.

## 사용 도구

- Codex (GPT-5 계열 코딩 에이전트)
- 주 작업은 메인 에이전트 중심으로 진행
- 필요할 경우 병렬 에이전트 사용 가능하도록 열어 두었지만, 이번 Phase 2 구현은 메인 흐름에서 직접 정리하고 구현했다

## 프롬프트 기록

### 1단계: 저장소와 공통 규약 파악
> `readme.md 를 읽어보고 common 폴더 내부를 먼저 파악한다음 내 담당인 member-dragon 폴더 내부를 한번 확인해봐`

**AI 응답 요약**: 루트 `README.md`, `common/`, `member-dragon/`를 읽고 현재 과제 범위와 폴더 상태를 정리했다. 공통 규약, 테스트 구조, `member-dragon/src`의 초기 상태까지 먼저 확인했다.  
**내 판단**: 구현 전에 공통 규약과 작업 폴더 상태를 먼저 맞추는 단계로 적절해서 그대로 채택했다.

### 2단계: 초기 구현 계획 리뷰
> `codex 구현을 위한 plan_codex.md 에 계획을 만들어놨어 먼저 한번 검토해보고 어떻게 이해했는지 나한테 알려줘`

**AI 응답 요약**: `plan_codex.md`를 검토하면서 `ResultSet` 메모리 소유권, 스키마 로드 실패 처리, statement 분리, INSERT 컬럼 처리의 문제를 짚고, 지금 계획이 어떤 구조를 목표로 하는지 설명했다.  
**내 판단**: 단순 요약이 아니라 실제 구현 전에 손봐야 할 설계 결함을 찾는 데 도움이 돼 채택했다.

### 3단계: 하네스 적용 + 1차 구현
> `검토결과에서 문제가 되는점은 수정을 하되 왜 수정했는지 뭐가 문제인지 기록해둬 . 그리고 하네스 프로그래밍을 적용하려고 하네스 스타터팩을 만들어서 폴더로 넣어놨는데 이것도 적용해서 하네스프로그래밍을 해보고 싶은데 서브 에이전트나 추가 에이전트가 필요하면 10개 이내로 추가해서 병렬로 움직여도 괜찮으니까 하네스 프로그래밍 적용해서 이번에 구현을 해보자`

**AI 응답 요약**: 하네스 문서와 로그를 갱신하면서 `member-dragon/src/` 구현, 테스트 통과, 기록 문서 업데이트까지 한 흐름으로 진행했다.  
**내 판단**: 이번 프로젝트의 핵심 지시였고, 단순 코드 생성이 아니라 기록과 검증까지 남기는 방향이 필요했기 때문에 그대로 채택했다.

### 4단계: 과제 요구사항 재확인 후 저장 방식 수정
> `요구사항이 이거라서 파일로 결과를 저장해야하는데 플랜이 이상했네`

**AI 응답 요약**: 기존 방향이 메모리 저장에 기울어 있었다는 점을 인정하고, `StorageOps` 구조는 유지한 채 `file_storage.c` 기반으로 바꿔 실제 파일 저장 과제 요구사항에 맞추자고 정리했다.  
**내 판단**: 테스트 통과보다 원 과제 요구사항이 우선이므로 파일 기반 저장소로 바로 수정하는 방향을 채택했다.

### 5단계: 실행 환경 정리
> `사용법도 하나 만들어줘 usage.md 파일에 만들어주면 좋을거 같아`

**AI 응답 요약**: `usage.md`를 만들고 빌드, 실행, 테스트 방법을 정리했다.  
**내 판단**: 구현만 있으면 재현이 불편하므로 실행 문서를 함께 남기는 쪽을 채택했다.

### 6단계: playground 실행 흐름 정리
> `지금 src 에서 make 했는데 실행용폴더 만드는게 낫나?`

**AI 응답 요약**: `src`는 빌드, `playground`는 실행/실험 용도로 분리하는 구조를 제안했고, 이후 실제 playground 폴더와 샘플 파일을 만들었다.  
**내 판단**: `.schema`, `.sql`, `.data`가 소스 폴더와 섞이지 않게 분리하는 편이 깔끔해서 채택했다.

### 7단계: 수동 테스트 보조 스크립트 추가
> `응 테스트 케이스들 스크립트 만들어줘`

**AI 응답 요약**: `playground`에서 공개/히든 테스트를 바로 돌려볼 수 있도록 `run_case.sh`, `run_suite.sh`를 만들었다.  
**내 판단**: 직접 실행과 기대값 비교를 반복해야 해서 재사용 가능한 스크립트를 두는 쪽을 채택했다.

### 8단계: 코드 이해를 위한 한글 주석 보강
> `생성 코드에 주석을 디테일하게 고등학생도 이해할수 있도록 쉽게 적어줘 함수 -> 무슨 역할인지 중요 내부 코드 -> 어떤 일을 하는지 정도로`

**AI 응답 요약**: 주요 함수와 핵심 코드 블록에 설명 주석을 달았다. 이후 요청에 맞춰 한글로 바꿨다.  
**내 판단**: AI로 만든 코드라도 핵심 로직을 설명할 수 있어야 하므로 교육용 주석 보강을 채택했다.

### 9단계: 코드 흐름 설명
> `이 프로그램이 돌아가는 과정을 따라가고 싶은데 디테일하게 하나씩 설명해줘`

**AI 응답 요약**: `main -> parser -> executor -> storage` 흐름으로 실제 SQL 파일이 어떻게 읽히고 저장/출력되는지 순서대로 설명했다.  
**내 판단**: 결과물 이해가 중요하다는 과제 취지와 맞아서 채택했다.

### 10단계: Phase 2 확장 계획 리뷰
> `중간까지 추가하는 계획을 새로 만들었어 plan_codex.md 확인해보고 검토 한번 해봐`

**AI 응답 요약**: 새 계획의 핵심 위험이 `WHERE`와 `ORDER BY`를 projection 이후에 처리하려는 점이라고 짚고, 그 구조로는 `SELECT name FROM users WHERE age > 22` 같은 쿼리가 깨진다고 리뷰했다.  
**내 판단**: 확장 구현 전에 구조를 바로잡지 않으면 기능은 붙어도 잘못 동작할 수 있어, 리뷰 내용을 반영해 계획을 수정하기로 했다.

### 11단계: Phase 2 실제 구현
> `응 네가 좀 수정하고 그 계획대로 구현까지 해줘 에이전트는 10개만 안넘기면 괜찮으니까 맘껏 써도 돼 사용량 많이 남았으니까 제대로 그리고 빠르게 구현해줘`

**AI 응답 요약**: `types.h`, `parser.c`, `file_storage.c`, `executor.c`, `main.c`를 확장해 `DELETE`, `UPDATE`, 단일 `WHERE`, `ORDER BY`, `LIMIT`, reordered `INSERT`를 구현했다. 핵심은 executor가 전체 row를 먼저 읽고 `WHERE -> ORDER BY -> LIMIT -> projection` 순서로 처리하도록 바꾼 점이다.  
**내 판단**: 기존 테스트를 깨지 않으면서 확장 기능을 안정적으로 넣을 수 있는 방향이라 채택했다.

### 12단계: 확장 기능 사용 문서와 키워드 테스트 파일 보강
> `사용법에 사용 가능한 키워드 전부 넣어주고 그리고 테스트 가능하도록 이번에 추가한 키워드가 포함된 테스트용 sql 파일도 playground에 만들어줘`

**AI 응답 요약**: `usage.md`에 지원 키워드/연산자를 전부 정리하고, `phase2_keywords.sql`과 기대 출력 파일을 만들어 `playground`에서 바로 돌려볼 수 있게 정리했다.  
**내 판단**: 새 기능이 많아졌기 때문에 "무엇을 지원하는지"와 "직접 검증하는 방법"을 같이 남기는 게 필요해서 채택했다.

### 13단계: 하네스 범위 문서 sync
> `하네스 프로그래밍 로직은 지켰어?`

**AI 응답 요약**: 하네스 흐름 자체는 지켰다고 설명하면서, 스타터팩 문서 일부가 아직 `INSERT`/`SELECT` 범위로 남아 있다는 점을 확인하고 `AGENTS.md`, `ARCHITECTURE.md`를 현재 Phase 2 범위에 맞게 sync했다.  
**내 판단**: 코드와 문서 범위가 어긋나 있으면 하네스 기록이 불완전하므로 문서 sync까지 마무리하는 방향을 채택했다.

### 14단계: 공유용 커밋/푸시
> `그럼 깃에 푸시 커밋 해줘`

**AI 응답 요약**: Phase 2 구현과 문서 변경을 `feat: add phase 2 SQL operations` 커밋으로 정리해 `feat/dragon` 브랜치에 푸시했다. 이어서 하네스 scope sync는 `docs: sync harness scope with phase 2`로 추가 커밋/푸시했다.  
**내 판단**: 구현 덩어리와 하네스 문서 sync를 분리한 커밋 구성이 리뷰와 추적에 더 적합해 채택했다.

## 정리

- AI는 단순 코드 생성보다도 설계 결함 탐지, 실행 흐름 정리, 문서화, 검증 자동화에 특히 도움이 되었다.
- 특히 Phase 2에서는 "projection 이후 WHERE/ORDER BY" 같은 잘못된 구조를 구현 전에 잡아낸 점이 가장 큰 기여였다.
- 최종적으로는 코드, 문서, playground 예제, 하네스 기록, 깃 이력까지 한 세트로 남기는 방향을 채택했다.
