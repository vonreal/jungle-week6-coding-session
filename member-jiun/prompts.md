# AI 활용 이력

## 사용 목적
빠르게 코드를 뽑는 것보다, 요구사항을 먼저 고정하고 테스트 기준을 명확히 하기 위해 AI를 활용했다.

## 작업 흐름 요약
1. 현재 폴더와 `common/` 문서를 함께 읽고, 과제 스펙과 채점 기준을 분리해 정리했다.
2. 확정 가능한 항목만 `요구사항 v1`로 잠그고, 미확정 항목은 별도 의사결정 목록으로 분리했다.
3. `CLI 입력 처리`를 1번 기능으로 잡고, 구현 전에 자동 테스트 케이스를 먼저 확장했다.
4. 인자 오류 정책(`ERROR: file open failed`, exit `1`, usage 미출력)을 확정한 뒤 테스트에 반영했다.

## 주요 프롬프트/의사결정 로그
1. 왜 먼저 요구사항을 고정해야 하는지 질문했다.
- AI 제안: 구현보다 스펙 고정이 먼저이며, `public` 통과 기준으로 단계 목표를 분리해야 한다.
- 내 판단: 채택. 현재 단계 DoD를 `public` 통과로 고정했다.

2. CSV/바이너리/확장성 관점에서 저장 포맷을 비교 요청했다.
- AI 제안: 지금은 CSV가 빠르지만, 확장성을 위해 스토리지 인터페이스 분리를 권장했다.
- 내 판단: 채택. 당장은 과제 우선, 구조는 확장 가능하게 설계하기로 했다.

3. CLI 테스트를 1개에서 엣지 케이스까지 확장 요청했다.
- AI 제안: 파일 없음 외에 디렉토리 경로, 권한 없음, 빈 파일, 인자 없음/과다를 포함하자고 제안했다.
- 내 판단: 채택. 총 6개 케이스로 확장하고 명세 문서를 추가했다.

## 현재 상태
- 요구사항 v1과 인자 오류 정책이 확정되었다.
- CLI 테스트 자산이 준비되었고, 다음 단계는 테스트를 통과하는 엔트리 구현이다.
