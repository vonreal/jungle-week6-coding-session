# AI 활용 이력

## 사용 목적
빠르게 코드를 뽑는 것보다 요구사항을 먼저 고정하고 테스트 기준을 명확히 하기 위해 AI를 활용했습니다.

## 작업 흐름 요약
1. 현재 폴더와 `common/` 문서를 함께 읽고 과제 스펙과 채점 기준을 분리해 정리했습니다.
2. 확정 가능한 항목만 `요구사항 v1`로 잠그고 미확정 항목은 별도 의사결정 목록으로 분리했습니다.
3. `CLI 입력 처리`를 1번 기능으로 정하고 구현 전에 자동 테스트 케이스를 먼저 확장했습니다.
4. 인자 오류 정책(`ERROR: file open failed`, exit `1`, usage 미출력)을 확정한 뒤 테스트에 반영했습니다.
5. `초기화`, `입력 처리`, `파싱`, `INSERT`, `SELECT`, `종료 처리`까지 기능 단위로 테스트 스위트를 먼저 만들었습니다.
6. 테스트 스위트를 기준으로 실제 `sql_processor`를 구현하고, 이후 `main.c` 단일 구조를 레이어별 모듈로 분리했습니다.

## 주요 프롬프트/의사결정 로그
1. 왜 먼저 요구사항을 고정해야 하는지 질문했습니다.
- AI 제안: 구현보다 스펙 고정이 먼저이며 `public` 통과 기준으로 단계 목표를 분리해야 합니다.
- 내 판단: 채택했습니다. 현재 단계 DoD를 `public` 통과로 고정했습니다.

2. CSV/바이너리/확장성 관점에서 저장 포맷 비교를 요청했습니다.
- AI 제안: 지금은 CSV가 빠르지만 확장성을 위해 스토리지 인터페이스 분리를 권장했습니다.
- 내 판단: 채택했습니다. 당장은 과제 우선으로 진행하고 구조는 확장 가능하게 설계하기로 했습니다.

3. CLI 테스트를 1개에서 엣지 케이스까지 확장 요청했습니다.
- AI 제안: 파일 없음 외에 디렉토리 경로, 권한 없음, 빈 파일, 인자 없음/과다를 포함하자고 제안했습니다.
- 내 판단: 채택했습니다. 총 6개 케이스로 확장하고 명세 문서를 추가했습니다.

4. 구현을 시작할 때 테스트 우선으로 진행할지 질문했습니다.
- AI 제안: 단계별 테스트를 먼저 만든 뒤, 그 테스트를 만족시키는 구현을 붙이는 방식이 안정적이라고 제안했습니다.
- 내 판단: 채택했습니다. `unit`, `cli`, `init`, `input`, `parse`, `insert`, `select`, `exit` 테스트 자산을 먼저 만들었습니다.

5. 구현 완료 후 `main.c`에 로직이 몰린 구조를 기능별로 분리 요청했습니다.
- AI 제안: `interfaces`, `application`, `domain`, `infrastructure`, `shared` 레이어로 나누고 `main.c`는 엔트리만 남기는 방향을 제안했습니다.
- 내 판단: 채택했습니다. 리팩터링 후에도 기존 테스트와 `public` 테스트를 다시 통과시키는 방식으로 검증했습니다.

## 현재 상태
- 요구사항 v1과 인자 오류 정책이 확정되었습니다.
- 테스트 자산은 `unit`, `CLI`, `초기화`, `입력 처리`, `파싱/실행 루프`, `INSERT`, `SELECT`, `종료 처리`까지 구성되었습니다.
- 실제 `sql_processor` 구현과 클린 아키텍처 기반 모듈 분리까지 완료되었습니다.
- 현재 기준 검증 범위는 자체 테스트 스위트와 `public` 테스트입니다.
