# 코드 기반 구두 검증 질문

이 문서는 현재 구현을 기준으로, 발표나 Q&A에서 바로 나올 수 있는 질문을 정리한 문서입니다.  
답변은 실제 코드 흐름을 기준으로 설명할 수 있도록 핵심 포인트 중심으로 적었습니다.

## 1. 전체 흐름

### Q. 프로그램은 어떤 순서로 동작합니까?

답변 포인트입니다.

1. `main()`이 `run_cli()`를 호출합니다.
2. `run_cli()`가 인자 개수를 검사하고 SQL 파일 경로를 넘깁니다.
3. `process_sql_file()`이 SQL 파일을 읽고 문장 단위로 분리합니다.
4. 비어 있지 않으면 스키마를 로드합니다.
5. 각 문장을 `INSERT` 또는 `SELECT`로 파싱합니다.
6. 파싱 성공 시 저장소 계층에서 실행합니다.
7. 에러가 나면 에러를 출력하고 다음 문장을 계속 처리합니다.
8. 마지막에 에러 발생 여부를 모아 종료코드를 결정합니다.

### Q. 왜 `main.c`를 얇게 유지했습니까?

답변 포인트입니다.

1. 진입점은 최대한 단순하게 두고 싶었습니다.
2. CLI, 실행 흐름, 파싱, 저장소 책임을 분리해야 읽기와 테스트가 쉬워집니다.
3. 그래서 `main()`은 `run_cli()` 호출만 하고 실제 로직은 각 레이어로 나눴습니다.

## 2. CLI와 입력 처리

### Q. 인자 오류를 왜 `ERROR: file open failed`로 통일했습니까?

답변 포인트입니다.

1. 이번 과제에서 허용된 표준 에러 메시지 집합을 우선했습니다.
2. 별도 `invalid arguments`를 추가하면 채점 기준과 어긋날 가능성이 있다고 판단했습니다.
3. 그래서 인자 오류도 파일을 정상적으로 열 수 없는 상황으로 통일했습니다.

### Q. 빈 SQL 파일은 왜 성공으로 처리합니까?

답변 포인트입니다.

1. 실행 가능한 문장이 하나도 없을 뿐, 파일 자체는 정상 입력으로 볼 수 있습니다.
2. 실제 코드도 문장 분리 후 결과가 0개면 바로 `0`을 반환합니다.
3. 불필요한 스키마 로드나 에러 출력도 하지 않도록 했습니다.

### Q. 입력 처리에서 어떤 엣지 케이스를 고려했습니까?

답변 포인트입니다.

1. 빈 줄과 공백만 있는 줄을 무시합니다.
2. 한 줄에 여러 SQL 문장이 있어도 `;` 기준으로 나눕니다.
3. 마지막 문장 뒤에 `;`가 없어도 처리합니다.
4. 따옴표 안에 있는 `;`는 문장 구분자로 보지 않습니다.
5. CRLF와 멀티라인 입력도 처리합니다.

## 3. 스키마와 검증

### Q. 왜 스키마를 미리 로드합니까?

답변 포인트입니다.

1. 존재하는 테이블인지 확인해야 합니다.
2. 존재하는 컬럼인지 검증해야 합니다.
3. `SELECT *`의 헤더 순서를 결정해야 합니다.
4. `INSERT` 시 컬럼 수와 타입 검증 기준이 필요합니다.

### Q. 스키마는 어떻게 읽습니까?

답변 포인트입니다.

1. 현재 버전은 `users.schema`, `products.schema` 두 파일을 필수로 읽습니다.
2. JSON 정식 파서를 쓰지 않고, 필요한 `"table"`, `"name"`, `"type"` 값을 문자열 탐색으로 추출합니다.
3. 과제 범위에는 충분하지만, 복잡한 JSON 구조까지 안전하게 처리하는 방식은 아닙니다.

### Q. 이 스키마 파싱 방식의 한계는 무엇입니까?

답변 포인트입니다.

1. 문자열 기반 파싱이라 형식이 많이 달라지면 취약합니다.
2. 일반적인 JSON parser보다 견고하지 않습니다.
3. 대신 의존성 없이 빠르게 구현하고 테스트하기에는 적합했습니다.

## 4. INSERT 처리

### Q. `INSERT`는 어떻게 검증합니까?

답변 포인트입니다.

1. 먼저 `INSERT INTO ... VALUES ...` 형태인지 검사합니다.
2. 테이블이 존재하는지 확인합니다.
3. 컬럼 목록과 값 목록을 각각 파싱합니다.
4. 컬럼 수와 값 수가 맞는지 확인합니다.
5. 컬럼명이 실제 스키마에 있는지 확인합니다.
6. 중복 컬럼 지정 여부를 확인합니다.
7. `int` 타입 컬럼은 숫자 리터럴인지 확인합니다.
8. 검증이 끝나면 스키마 컬럼 순서에 맞춰 `plan->values`에 재배치합니다.

### Q. 왜 컬럼 순서 재배치가 필요합니까?

답변 포인트입니다.

1. 사용자가 `INSERT INTO users (age, name, major)`처럼 순서를 바꿔서 넣을 수 있기 때문입니다.
2. 저장은 테이블 스키마 순서대로 일관되게 해야 이후 `SELECT`가 단순해집니다.

### Q. 값 안의 공백이나 쉼표는 어떻게 처리합니까?

답변 포인트입니다.

1. 작은따옴표로 감싼 값은 문자열로 취급합니다.
2. 문자열 안의 공백과 쉼표는 값의 일부로 유지합니다.
3. 저장할 때는 탭, 개행, 역슬래시 같은 문자를 이스케이프해서 파일 포맷을 깨지 않게 합니다.

## 5. SELECT 처리

### Q. `SELECT *`와 `SELECT name, age`는 어떻게 다르게 처리합니까?

답변 포인트입니다.

1. `SELECT *`이면 스키마의 전체 컬럼 인덱스를 그대로 사용합니다.
2. 특정 컬럼 목록이면 각 컬럼명을 스키마에서 찾아 인덱스로 변환합니다.
3. 실행 단계에서는 저장된 한 줄을 컬럼 배열로 복원한 뒤, 필요한 인덱스만 골라 출력합니다.

### Q. 결과가 0건이면 왜 헤더도 출력하지 않습니까?

답변 포인트입니다.

1. 이번 과제 규약을 우선했습니다.
2. 현재 구현은 데이터 행이 하나도 없으면 출력하지 않도록 맞춰져 있습니다.
3. 일반 DB와는 다를 수 있지만, 과제 요구사항과 테스트 정합성을 우선했습니다.

## 6. 저장소 설계

### Q. 왜 CSV가 아니라 현재 포맷을 썼습니까?

답변 포인트입니다.

1. CSV는 값 안의 쉼표 처리 때문에 오히려 파싱 부담이 늘어납니다.
2. 현재 구현은 탭 구분 기반 텍스트 포맷이라 컬럼 분리가 단순합니다.
3. 대신 탭, 개행, 역슬래시를 이스케이프해서 값 손실을 막았습니다.
4. 사람이 읽을 수 있고, 과제 범위에서 구현 난이도도 적절했습니다.

### Q. 왜 SQLite를 바로 쓰지 않았습니까?

답변 포인트입니다.

1. 이번 과제는 SQL 처리기의 핵심 흐름을 직접 구현하는 것이 중심이었습니다.
2. SQLite를 쓰면 저장소 안정성은 높아지지만, 파싱과 실행 흐름을 직접 보여주는 비중이 줄어듭니다.
3. 현재는 파일 기반 저장을 직접 구현하고, 구조는 추후 저장소 교체가 가능하도록 분리했습니다.

## 7. 에러와 종료 처리

### Q. 왜 에러가 나도 다음 문장을 계속 실행합니까?

답변 포인트입니다.

1. 과제 요구사항이 그렇기 때문입니다.
2. 실제 코드도 각 문장마다 실패 시 `had_error`만 기록하고 `continue`합니다.
3. 마지막에 한 번만 전체 종료코드를 계산합니다.

### Q. 종료코드는 어떻게 결정합니까?

답변 포인트입니다.

1. 문장 처리 중 에러가 한 번이라도 나오면 `had_error = 1`로 기록합니다.
2. 모든 문장을 돈 뒤, `had_error`가 있으면 `1`, 없으면 `0`을 반환합니다.

### Q. 한 문장에서 에러가 여러 개면 어떻게 합니까?

답변 포인트입니다.

1. 현재 구현은 대표 에러 하나만 출력합니다.
2. 파싱 또는 검증의 첫 실패 지점에서 에러 메시지를 결정하고 종료합니다.
3. 과제 요구사항의 단순하고 일관된 에러 처리에 맞춘 선택입니다.

## 8. 한계와 확장

### Q. 현재 구현이 보장하지 않는 것은 무엇입니까?

답변 포인트입니다.

1. 동시 쓰기 충돌 제어
2. 트랜잭션 rollback
3. 장애 이후 durability 보장
4. 백업 및 시점 복구
5. 권한, 암호화, 감사 로그

### Q. 그럼 이 구조에서 다음 확장은 어떻게 합니까?

답변 포인트입니다.

1. `domain`은 SQL 규칙과 검증에 집중합니다.
2. `infrastructure`는 실제 저장 방식만 담당합니다.
3. 그래서 저장소 계층을 SQLite나 다른 방식으로 교체하는 확장이 상대적으로 쉽습니다.
4. 기능 확장은 `WHERE`, `UPDATE`, `DELETE` 순으로 붙일 수 있습니다.
