# 설계 판단 기록

## 목표
이번 구현은 기능 확장보다 `public` 테스트 100% 통과를 우선한다.  
그래서 복잡한 SQL 엔진보다는, 스펙에 정확히 맞는 안정적인 처리기를 빠르게 완성하는 방향으로 설계한다.

## 왜 이렇게 설계했는가
1. 현재 단계는 개인 구현 검증 단계라서 히든/확장 기능보다 공개 테스트 정합성이 중요하다.
2. 에러 메시지와 종료코드가 채점 포인트라서 동작보다 출력 규약을 먼저 고정해야 한다.
3. 저장소를 과도하게 복잡하게 만들면 파싱/실행 로직 검증이 늦어지므로 파일 기반 단순 구조를 채택한다.

## 어떻게 구현할 것인가
1. 지원 범위를 `INSERT`, `SELECT`로 제한한다.
2. 실행 단위는 `./sql_processor <sql_file>`로 고정하고, SQL 파일을 순차 처리한다.
3. 에러는 문장당 1개만 `stderr`에 표준 문구로 출력하고 다음 문장을 계속 실행한다.
4. 출력은 `SELECT`에서만 CSV 헤더+행 형태로 내보내고, 결과 0건이면 출력하지 않는다.
5. 종료코드는 전체 실행 결과를 집계해 결정한다(에러 1건 이상이면 `1`).

## 요구사항 v1 (확정 요약)
1. 현재 단계 합격 기준은 `public` 테스트 100% 통과다.
2. `CREATE TABLE`은 구현하지 않고, 스키마/테이블은 사전 존재를 전제로 한다.
3. 데이터 저장은 파일 기반으로 처리한다.
4. 키워드는 대소문자를 구분하지 않는다.
5. 표준 에러 문구는 아래 4개만 사용한다.
- `ERROR: invalid query`
- `ERROR: table not found`
- `ERROR: column count does not match value count`
- `ERROR: file open failed`

## 인자 오류 정책 (확정)
1. 인자 없음/인자 과다 모두 `ERROR: file open failed`를 `stderr`에 출력한다.
2. 종료코드는 `1`로 고정한다.
3. 추가 usage 문구는 출력하지 않는다.
