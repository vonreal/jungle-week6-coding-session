# 설계 판단 기록

## 목표
이번 구현은 기능 확장보다 `public` 테스트 100% 통과를 우선합니다.  
그래서 복잡한 SQL 엔진보다는 스펙에 정확히 맞는 안정적인 처리기를 빠르게 완성하는 방향으로 설계합니다.

## 왜 이렇게 설계했는가
1. 현재 단계는 개인 구현 검증 단계이므로 히든/확장 기능보다 공개 테스트 정합성이 중요합니다.
2. 에러 메시지와 종료코드가 채점 포인트이므로 동작보다 출력 규약을 먼저 고정해야 합니다.
3. 저장소를 과도하게 복잡하게 만들면 파싱/실행 로직 검증이 늦어지므로 파일 기반 단순 구조를 채택합니다.

## 어떻게 구현할 것인가
1. 지원 범위를 `INSERT`, `SELECT`로 제한합니다.
2. 실행 단위는 `./sql_processor <sql_file>`로 고정하고 SQL 파일을 순차 처리합니다.
3. 에러는 문장당 1개만 `stderr`에 표준 문구로 출력하고 다음 문장을 계속 실행합니다.
4. 출력은 `SELECT`에서만 CSV 헤더+행 형태로 내보내고 결과 0건이면 출력하지 않습니다.
5. 종료코드는 전체 실행 결과를 집계해 결정합니다(에러 1건 이상이면 `1`입니다).

## 요구사항 v1 (확정 요약)
1. 현재 단계 합격 기준은 `public` 테스트 100% 통과입니다.
2. `CREATE TABLE`은 구현하지 않고 스키마/테이블은 사전 존재를 전제로 합니다.
3. 데이터 저장은 파일 기반으로 처리합니다.
4. 키워드는 대소문자를 구분하지 않습니다.
5. 표준 에러 문구는 아래 4개만 사용합니다.
- `ERROR: invalid query`
- `ERROR: table not found`
- `ERROR: column count does not match value count`
- `ERROR: file open failed`

## 인자 오류 정책 (확정)
1. 인자 없음/인자 과다 모두 `ERROR: file open failed`를 `stderr`에 출력합니다.
2. 종료코드는 `1`로 고정합니다.
3. 추가 usage 문구는 출력하지 않습니다.

## 저장 포맷 결정
현재 구현은 각 테이블을 `<table>.data` 파일로 저장하는 커스텀 텍스트 포맷을 사용합니다.

이렇게 정한 이유입니다.
1. CSV처럼 사람이 읽을 수 있는 장점을 일부 유지하면서도, 값 내부 쉼표를 안전하게 보존할 수 있습니다.
2. 현재 과제 범위에서는 인덱스나 복잡한 바이너리 포맷보다 구현과 검증 속도가 더 중요합니다.
3. 특수문자를 이스케이프하는 방식으로 `SELECT` 출력 형식과 내부 저장 형식을 분리할 수 있습니다.

## 구조 분리 결정
초기에는 `main.c` 하나에 모든 로직이 들어가는 형태로 구현했지만, 이후 레이어별로 분리했습니다.

이렇게 바꾼 이유입니다.
1. 테스트 기준으로 기능이 늘어나면서 한 파일에 모든 책임이 모이면 읽기와 검증이 어려워졌습니다.
2. CLI, 실행 흐름, 파싱, 저장소, 문자열 유틸은 서로 변경 이유가 다르므로 모듈로 나누는 편이 유지보수에 유리합니다.
3. 발표에서 “어디서 무엇을 담당하는지” 설명하기 쉬운 구조가 필요했습니다.

현재 역할 분리는 아래와 같습니다.
1. `interfaces`: CLI 진입점 처리
2. `application`: 전체 실행 오케스트레이션
3. `domain`: 스키마/파싱/검증 규칙
4. `infrastructure`: 파일 읽기/쓰기와 저장소 실행
5. `shared`: 문자열, 벡터, 에러 출력 공통 유틸

## 빈 SQL 파일 처리 결정
빈 SQL 파일은 스키마 로드 전에 바로 성공 종료(`0`)하도록 두었습니다.

이렇게 정한 이유입니다.
1. 실행할 SQL 문장이 없으면 처리할 작업도 없으므로 no-op으로 보는 편이 자연스럽습니다.
2. CLI 테스트에서 빈 파일은 정상 실행으로 다루고 있습니다.
3. 불필요한 스키마 의존성을 줄여 입력 처리와 초기화 책임을 더 명확히 나눌 수 있습니다.
