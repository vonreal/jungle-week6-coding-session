# 삽질 로그

## 이슈 1: `main.c` 단일 파일 구조가 너무 커진 문제
- **상황**: 테스트를 모두 만족하는 구현까지는 빠르게 만들었지만, 파싱/저장/CLI/종료 처리 로직이 한 파일에 몰려 읽기와 설명이 어려워졌습니다.
- **원인**: 초기에 구현 속도를 우선하면서 기능을 한 파일에 계속 쌓았기 때문입니다.
- **AI가 준 답**: 먼저 테스트로 안정성을 확보한 뒤, 책임 기준으로 `interfaces`, `application`, `domain`, `infrastructure`, `shared`로 분리하자고 제안했습니다.
- **실제 해결**: 기존 로직을 레이어별 모듈로 옮기고 `main.c`는 엔트리만 남겼습니다. 이후 자체 테스트와 `public` 테스트를 다시 돌려 회귀를 확인했습니다.
- **배운 것**: 빠른 구현과 좋은 구조는 동시에 달성하기 어렵지만, 테스트가 있으면 나중에 안전하게 구조를 바꿀 수 있습니다.

## 이슈 2: 테스트 스크립트의 `trap`과 `set -u` 충돌
- **상황**: 테스트 러너 실행 중 `tmp: unbound variable` 오류가 발생했습니다.
- **원인**: 함수 내부 `local tmp`와 `trap ... RETURN`이 `set -u`와 결합되면서 반환 시점에 변수를 안전하게 참조하지 못했습니다.
- **AI가 준 답**: 러너를 깨끗한 상태로 복구한 뒤, `tmp`를 함수 스코프 local 대신 직접 할당하는 방식으로 단순화하자고 제안했습니다.
- **실제 해결**: 각 러너에서 `tmp="$(mktemp -d)"` 패턴으로 통일해 `trap`이 안정적으로 동작하도록 수정했습니다.
- **배운 것**: 셸 스크립트에서는 작은 스코프 차이도 `set -u`와 만나면 예민해질 수 있으므로, 테스트 인프라는 단순성이 중요합니다.

## 이슈 3: 디렉토리 경로가 파일처럼 처리되던 문제
- **상황**: CLI 테스트 중 디렉토리 경로를 입력했을 때 `file open failed`가 아니라 성공처럼 처리되는 경우가 있었습니다.
- **원인**: 파일 읽기 로직이 단순 `fopen`만 사용해 경로 타입을 먼저 검증하지 않았기 때문입니다.
- **AI가 준 답**: `stat`으로 regular file 여부를 먼저 확인한 뒤 파일을 읽자고 제안했습니다.
- **실제 해결**: `read_entire_file`에서 `stat`과 `S_ISREG` 검사를 추가해 디렉토리 입력을 즉시 실패 처리하도록 수정했습니다.
- **배운 것**: 파일 입출력은 “열 수 있는가”만 보지 말고 “무엇을 열고 있는가”도 먼저 확인해야 합니다.
