# member-jihyun 프로젝트 체크리스트

## 1. 환경 및 빌드
- [x] `member-jihyun/src`에서 `make`가 경고/에러 없이 완료된다.
- [x] 필요 시 `make clean` 후 재빌드해도 동일하게 성공한다.
- [x] 실행 파일 `sql_processor`가 생성된다.

## 2. 기본 실행 동작
- [x] 실행 인자가 없거나 잘못되면 에러 메시지가 출력된다.
- [x] SQL 파일 1개를 인자로 받았을 때 정상 실행된다.
- [x] 여러 SQL 문장을 `;` 기준으로 순차 실행한다.

## 3. 파싱 및 문법 처리
- [x] `INSERT INTO ... (...) VALUES (...)` 구문이 정상 파싱된다.
- [x] `SELECT ... FROM ...` 구문이 정상 파싱된다.
- [x] `SELECT * FROM ...` 구문이 정상 파싱된다.
- [x] 대소문자 혼용 SQL(예: `SeLeCt`)을 정상 처리한다.
- [x] 따옴표/이스케이프(`''`)가 포함된 값 처리가 정상 동작한다.

## 4. 스키마/검증 로직
- [x] `.schema` 파일을 읽어 테이블/컬럼 정보를 로드한다.
- [x] 존재하지 않는 테이블은 `table not found` 계열 에러를 반환한다.
- [x] 컬럼 개수와 값 개수 불일치 시 에러를 반환한다.
- [x] 존재하지 않는 컬럼 참조 시 에러를 반환한다.

## 5. 저장소(Page + Buffer Cache)
- [x] 데이터가 `{table}.data` 파일에 기록된다.
- [x] 페이지 용량 초과 시 새 페이지가 생성된다.
- [x] dirty 페이지 flush가 정상 반영된다.
- [x] 종료 시 flush가 호출되어 데이터 유실이 없다.

## 6. 출력 형식
- [x] `SELECT` 결과 헤더가 CSV 형태(`col1,col2`)로 출력된다.
- [x] 각 행이 CSV 형태로 출력된다.
- [x] 에러 메시지가 과제 형식(`ERROR: ...`)으로 출력된다.

## 7. 테스트 검증
- [x] `common/tests/public` 전체를 실행한다.
- [x] public 결과가 6/6 PASS인지 확인한다.
- [x] 실패 케이스가 있으면 원인/수정 내역을 `trouble.md`에 기록한다. (이번 실행은 실패 0건으로 해당 없음)

## 8. 문서화
- [x] `decisions.md`에 핵심 설계 선택 근거를 최신 상태로 유지한다.
- [x] `prompts.md`에 실제 AI 요청/응답 요약을 누락 없이 기록한다.
- [x] `trouble.md`에 문제-원인-해결-재발방지 형식으로 정리한다.
- [x] 코드 주석은 함수 단위로 역할이 드러나게 유지한다.

## 9. 발표/시연 준비
- [ ] "입력 -> 파싱 -> 실행 -> 저장 -> 출력" 흐름을 1분 내 설명 가능하다.
- [ ] public 테스트 실행 장면을 재현할 수 있다.
- [ ] 예상 질문(왜 CSV 파일이 없는지, `.data` 구조, `.o/.d` 의미)에 답변 가능하다.

## 10. 제출 전 최종 점검
- [ ] 불필요한 임시 파일이 없다.
- [ ] 필수 문서 3종(`decisions.md`, `prompts.md`, `trouble.md`)이 채워져 있다.
- [ ] 최종 테스트 통과 상태를 다시 확인했다.
- [ ] 커밋 메시지가 변경 의도를 명확히 설명한다.
