## Step 2. 개인 발표 (1인당 5분, 전체 20~25분)

한 명씩 돌아가면서. 이 세 가지만 말해:

### 2-1. decisions.md 공유 (2분)
- 저장 포맷 뭘로 했고 왜
- 파서 어떻게 구조 잡았고 왜
- 에러 처리 어디까지 했고 왜

### 2-2. trouble.md 공유 (2분)
- 가장 크게 막힌 부분
- AI가 뭘 틀렸는지
- 어떻게 해결했는지

### 2-3. 핵심 코드 보여주기 (1분)
- 본인 코드에서 가장 중요한 함수 1~2개만 화면에 띄워
- "이게 내 파서의 핵심이다" 수준으로 짧게

---

## Step 3. 비교 토론 (20분)

발표 다 끝나면, 아래 질문으로 토론:

### 구조 비교
- 파일 몇 개로 나눴어? (1개 vs 5개 vs ?)
- 함수 몇 개야?
- 제일 긴 함수가 몇 줄이야?

### 파서 비교
- strtok 썼어? 직접 파싱했어?
- 따옴표 안 쉼표(히든 06번) 어떻게 처리했어?
- 대소문자(공개 05번) 어떻게 처리했어?

### AI 활용 비교
- prompts.md 서로 보기
- 한 번에 전체 시켰어? 단계별로 시켰어?
- AI 출력 그대로 썼어? 얼마나 고쳤어?
- 제일 유용했던 프롬프트는?

### 삽질 비교
- 가장 오래 걸린 부분이 어디야?
- 그거 다른 사람은 어떻게 피했어?

## Step 4. 즉석 스트레스 테스트 (15분)

다 통과했으니까, 새로운 엣지 케이스를 그 자리에서 만들어서 돌려봐.
누가 먼저 깨지는지 보는 거야.

시도해볼 것들:
```sql
-- 세미콜론 없는 SQL
INSERT INTO users (name, age, major) VALUES ('김민준', 25, '컴퓨터공학')

-- 빈 문자열
INSERT INTO users (name, age, major) VALUES ('', 25, '컴퓨터공학');

-- SQL 키워드가 값에 포함
INSERT INTO users (name, age, major) VALUES ('SELECT', 25, 'FROM');

-- 숫자에 음수
INSERT INTO products (name, price, category) VALUES ('할인상품', -5000, '세일');

-- 아주 긴 문자열
INSERT INTO users (name, age, major) VALUES ('가나다라마바사아자차카타파하가나다라마바사아자차카타파하', 25, '컴퓨터공학');

-- 같은 테이블에 SELECT 3번 연속
SELECT * FROM users;
SELECT name FROM users;
SELECT * FROM users;

-- 완전히 잘못된 SQL
THIS IS NOT SQL;

-- 빈 파일
(아무 내용 없는 .sql 파일)
```

각자 이걸 .sql 파일로 만들어서 돌려보고, 결과를 비교.
깨진 케이스가 있으면 → "이걸 고치려면 어디를 바꿔야 해?"를 같이 논의.

## Step 5. 베스트 선정 (5분)

아래 기준으로 투표:

- 코드 가독성이 제일 좋은 건?
- 엣지 케이스를 가장 많이 버틴 건?
- 구조가 가장 확장하기 쉬운 건?

→ 하나를 베이스로 선정. Phase 4에서 이걸 기반으로 최종본 만들기.
